3. Frontend Stack

3.1 React 18
Usage:
•	All UI components (Navbar, Home, GameShowcase, GameView, AnalyticsPreview)
•	State management via useState/useEffect hooks
•	ML integration via React Context (MLProvider)
•	AnimatePresence for page transitions

Why React:
•	Component reusability across pages
•	Rich ecosystem (Framer Motion, GSAP integration)
•	Virtual DOM for efficient re-renders
•	Large community and documentation

3.2 Vite 5
Usage:
•	Development server with hot module replacement (HMR)
•	Production build with esbuild minification
•	API proxy (frontend :5173 → backend :8000)
•	Manual chunk splitting for code optimization

Build Performance:
•	Dev server: instant startup
•	Production build: ~2-6 seconds
•	Zero warnings policy
•	Chunk splitting: vendor (React, Framer Motion) + game engines

Why Vite:
•	20x faster than webpack for dev
•	Native ES module support
•	Built-in proxy for API forwarding
•	Excellent build optimization

3.3 Framer Motion
Usage:
•	Page transitions (fade, slide)
•	Component mount/unmount animations
•	Result modal spring animations
•	Performance bar animated fills

3.4 GSAP (GreenSock)
Usage:
•	Animated number counters in analytics
•	Complex timeline-based animations
•	Scroll-triggered effects

3.5 CSS
Usage:
•	CSS custom properties (--color-primary, --bg-surface, etc.)
•	Dark/light theme toggle via class switching
•	Responsive layouts with flexbox/grid
•	Glassmorphism and gradient effects
•	Micro-animations for hover/focus states

4. Game Engine Stack

4.1 Vanilla JavaScript (ES6+)
Usage:
•	10 independent game engine classes
•	Each game: init(), render(), cleanup() lifecycle
•	Event bus pattern for game:move, game:end events
•	Deterministic state machines

Why Vanilla JS (not React components):
•	Game logic must be deterministic and framework-independent
•	Direct DOM manipulation for canvas/grid rendering
•	Easy to test and replay
•	React wraps them via ref containers (GameView.jsx)

Game Architecture:
  constructor(eventBus) — receives communication channel
  init()               — resets game state
  render(state, el)    — renders into DOM element
  cleanup()            — removes listeners, clears state
  eventBus.emit()      — broadcast move/end events

5. ML Stack (Frontend)

5.1 FeatureExtractor.js
•	Records every move: timestamp, decision time, correctness
•	Extracts 15+ aggregate features on session end
•	Calculates: accuracy, consistency (CV), patterns, optimal play rate
•	Game-specific feature extraction logic

5.2 MLClient.js
•	Communicates with Python API via fetch()
•	Caching: avoids redundant API calls for identical features
•	Fallback: rule-based predictions when API is offline
•	Timeout handling: 5s abort on slow responses
•	Health checks: pings /api/ml/health on init

5.3 MLProvider.jsx (React Context)
•	Provides singleton MLClient + FeatureExtractor to all components
•	Manages session history in localStorage
•	Exposes hooks: startSession(), recordMove(), endSessionAndPredict()

6. ML Stack (Backend — Python)

6.1 FastAPI
Usage:
•	REST API endpoints for ML inference
•	Pydantic models for automatic request validation
•	Async endpoint handlers
•	Auto-generated Swagger docs at /docs

Why FastAPI (replaced Flask):
•	~40x higher throughput (async vs sync)
•	Automatic request validation via type hints
•	Built-in API documentation
•	Native async/await support

6.2 Uvicorn (ASGI Server)
Usage:
•	Runs FastAPI application
•	Multi-worker mode: uvicorn server:app --workers 4
•	Hot reload in development

Capacity:
•	Single worker: ~2,000 req/s
•	4 workers: ~8,000 req/s
•	Sufficient for 10,000 concurrent users

6.3 scikit-learn
Usage:
•	RandomForestClassifier for skill tier prediction
•	RandomForestRegressor for performance estimation
•	train_test_split, cross_val_score for evaluation
•	Models serialized as .pkl files

6.4 NumPy + Pandas
•	Feature array construction
•	Statistical calculations
•	Data normalization (min-max scaling)

7. Scalability Stack

7.1 Current Patterns (In-Memory)

Pattern             Implementation        Purpose
Cache-Aside         TTLCache (60s TTL)    Avoid redundant ML inference
Token Bucket        RateLimiter class     Prevent API abuse (100/min/IP)
Singleton           ModelPool class       Load models once, reuse forever
Gzip Middleware     FastAPI middleware     Compress responses >500 bytes
Request Timing      Custom middleware     Track response times

7.2 Future Patterns (Planned)

Redis               Distributed caching   Multi-instance cache sharing
PostgreSQL          Relational database   Persistent session storage
Celery + Redis      Task queue            Async batch processing
Nginx               Reverse proxy         Load balancing across workers
Docker              Containerization      Consistent deployment

10. Getting Started (for Contributors)

Prerequisites:
•	Node.js 18+
•	Python 3.9+

Setup:
  # Clone
  git clone https://github.com/BandameediJayanth/Dual-Mind.git
  cd Dual-Mind

  # Frontend
  npm install
  npm run dev              # → http://localhost:5173

  # Backend (separate terminal)
  cd python
  python -m venv venv
  .\venv\Scripts\activate  # Windows
  pip install -r requirements.txt
  uvicorn server:app --port 8000 --reload

  # Build for production
  npm run build            # → dist/ folder

11. Constraints & Trade-offs

Decision                 Rationale
────────────────────────────────────────────────────────────
React (not vanilla HTML)  Component reuse, animation libraries, context API
FastAPI (not Flask)       40x throughput, async, auto-docs
Random Forest (not NN)    Explainable, works with small synthetic data
Synthetic data (not real) No user base yet — planned collection pipeline
In-memory cache (not Redis) Zero-config for dev, Redis planned for prod
Vanilla JS engines (not React) Deterministic, framework-independent, testable
